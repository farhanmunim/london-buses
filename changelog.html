v2.9 May 1, 2026
Full route-card record now lands on every weekly Supabase snapshot

Migration 0009_route_snapshot_full_record.sql adds 20 columns to route_snapshots so each weekly row carries the full route-card record without needing JOINs to tenders / tender_programme / route_performance at query time. Three groups: reliability snapshot (service_class, ewt_minutes, on_time_percent, perf_period); last/current tender (previous_operator, last_award_date, last_cost_per_mile, tender_award_count, number_of_tenderers, was_joint_bid, contract_term_years, awarded_propulsion, awarded_deck, prev_awarded_propulsion, prev_awarded_deck); and next/upcoming tender (next_tender_start, next_tender_year, extension_eligible, next_award_propulsion, next_award_deck).

pushRouteSnapshots() extended to write all 20 new columns from the existing route_classifications.json derivations — no new fetch step or schema work elsewhere. First push under 0009 populated 733/747 routes with award data, 665/747 with reliability data, 694/747 with contract length. Three new indexes (service_class, previous_operator, last_award_date) enable single-table trend queries — a "worst-performers-vs-MPS" board is now one SELECT with no JOINs.

Senior-engineer review confirmed the rest of the stack is structurally sound: 16-step orchestrator matches disk + npm scripts; soft-fail vs hard-fail correctly assigned; workflow secrets / permissions / force-add list aligned; build outputs healthy; frontend renders all tooltips with zero JS errors. Dead TYPE_MAP constant removed from route-detail.js.

v2.8 May 1, 2026
Per-route contractual EWT/OTP/mileage standards on every route card

Per-route Minimum Performance Standards (MPS) now flow through the whole stack. A new fetch-route-mps.js step parses the per-route TfL QSI PDFs at bus.data.tfl.gov.uk/boroughreports/routes/performance-route-{ID}.pdf and extracts three contractual benchmarks per route: EWT MPS (high-frequency, observed 0.7–1.4 min), OTP MPS (low-frequency, observed 74–90%), and Mileage MPS (98 or 99%). The values genuinely vary route-by-route within a service class — e.g. route 122 (EWT 1.20 min) and route EL2 (EWT 0.70 min) are both high-frequency but have very different contractual thresholds. 662 routes parsed cleanly; 85 (mostly school) have no published MPS PDF and cache as 404.

Surfaced on the route card in the Tender · Current contract section as EWT standard / OTP standard / Mileage standard rows (hidden when no MPS data). The Route-section reliability KPI tile also gets an inline actual / standard reading (e.g. 1.6 / 1.0) so pass/fail is readable at a glance.

Pushed to Supabase via migration 0008_route_mps.sql: three new columns on route_snapshots (ewt_mps_minutes, otp_mps_percent, mileage_mps_percent) so trend queries can see the benchmark drift over time as contracts renew. The XLSX export Routes sheet picks them up automatically.

Sticky-cache pipeline. MPS values are contract-defined and only move at tender renewal, so the cache uses a 28-day TTL plus per-route Last-Modified HEAD short-circuit. Cold-start week parses ~700 PDFs at 4 RPS (~3 min); steady-state weekly runs are mostly HEAD-only and finish in seconds. Per-run cap of 1,000, periodic flush every 50, SIGTERM-safe.

Route-card UX polish. Tender section restructured into Current contract / Previous contract (the Next contract block was removed since nothing in it has actually been awarded yet). Labels tightened: Last awarded → Awarded on, Total awards → Times tendered, Vehicle make / Vehicle model → Make / Model, Contract term → Length. Deck values render as DD / SD; the frequency tile shows just H / L. The route-type chip moved next to the route number and only renders for non-default classes (24h / Night / School / Prefix). A new MPS KPI tile sits next to EWT/OTP so contractual targets and actuals read side-by-side. Tender-history operator names roll up to the parent group (Docklands / Blue Triangle / Metrobus → Go-Ahead, Selkent / East London → Stagecoach, London United / Quality Line → RATP).

Custom hover tooltips on every route-card label (js/tooltip.js) replacing the native title tooltip so we control timing and look. Labels get a faint dashed underline as the "more info on hover" cue; the popup is a single shared element appended to <body>, position-tracked above its anchor with viewport-edge clamping and a 120 ms show delay. Tooltip copy is short "<metric>, from <source>" lines so each label answers what the value is and where it comes from at a glance.

v2.7 Apr 30, 2026
Tender data on every route card, three pipeline short-circuits, and contributors

Tender award history is now surfaced on every route card. The placeholder rows for Previous operator, Contract expires, and Contract value are wired up. build-classifications.js joins data/source/tenders.json (~2,500 historical awards going back to 2003) with data/source/tender-programme.json (the LBSL programme PDFs) to compute per-route previousOperator, lastAwardDate, lastCostPerMile (£/mile, directly comparable across routes), nextTenderStart and nextTenderYear. Combined route IDs like 341/N341 automatically feed both the day route and its night sibling. "Previous operator" walks the sorted award list to find the most recent earlier operator that genuinely differs from the current incumbent, so a route re-awarded twice to the same operator still shows its real predecessor.

Tender data pushed to Supabase with four derived columns on public.tenders (propulsion_type, is_joint_bid, vehicles_basis, previous_operator) and two on public.tender_programme (propulsion_type, previous_operator). All derivations run automatically in push-to-supabase.js from the existing notes / vehicle-type / award-date fields; data/tender-overrides.json applies any manual corrections.

Three pipeline short-circuits stop the weekly run from re-doing work upstream hasn't changed. fetch-data.js lists S3, compares the latest ZIP date against data/geometry-source.json::zipDate, and skips the ~10 MB download + 700 file rewrites when unchanged (≈95% of weeks). fetch-route-performance.js sends a HEAD first and skips the PDF parse if Last-Modified matches the cached pdfModifiedAt — the QSI PDF only republishes every ~4 weeks so most runs now exit in milliseconds. fetch-tender-programme.js does a per-year HEAD; closed financial years are copied forward unchanged, dropping 10 PDF parses to 1 in steady state. All three accept --force for manual re-runs.

Truthful nulls in the build. When a source file loads cleanly but a route has no entries, the build writes null instead of falling back to last-known-good — otherwise stale values persist forever. Last-known-good now only kicks in when the source file itself is missing (i.e. the upstream scrape failed).

About → Contributors. Daniel Plumb, Mark Leonard-Adoko, and Ross Levine added under the Developer section.

v2.6 Apr 28, 2026
Binary frequency rule, BZL→EV fix, and an automated tier-3 headway fallback

Frequency is now a binary classification. A route is High if it operates 5 or more buses per hour (≤12 min headway), and Low otherwise. The old three-band scheme (Low / Regular / High at ≤6 / 7–15 / >15 min) is gone — across the bin function, the route-card KPI, the filter pills, the schema docs, and every data file. SL7 is manually pinned to High via route-overrides.json since its scraped 15-min headway would otherwise sort it Low.

Propulsion correction for 11 routes. Stagecoach London and Metroline encode their MCV eVoSeti EV buses with the fleet code BZL in details.htm (e.g. BZL (dd) 10.9m/MCV 2D). The body string alone has no EV marker and the prefix is stripped during cleanup, so the regex was defaulting to diesel. Adding \bBZL\b to the EV pattern correctly classifies D7, D8, 58, 187, 228, 251, 276, 314, 316, 384, 487 as Electric.

The pipeline now reads the headway columns straight from details.htm. A new representativeHeadway() in fetch-route-details.js parses the Mon-Sat / Sunday / evening cells (handling night-route layout where headways sit just before the date, and skipping school routes whose columns hold endpoint names instead of numbers). The value flows through route_details.json as headwayMin per route. build-classifications.js uses it as a tier-3 frequency fallback when TfL /Line/<id>/Timetable (tier 1) and times/<id>.htm (tier 2) both yield null — same ≤12 cutoff so the rule is identical across every tier. Every weekly refresh now reproduces the manual rebin automatically.

v2.5 Apr 24, 2026
A new Network Overview with operator and garage drawers

The right-hand panel is now a proper Network Overview. Four KPI cards (Routes, Operators, Garages, PVR) sit above a per-operator breakdown with Routes %, PVR % and EV % — tap any row and the drawer slides in with routes, PVR, a PVR-weighted Fleet Mix stacked bar and the operator's garages. Each garage opens its own drawer with the same shape plus a View all routes operated here CTA that loads those routes onto the map and stacks route cards in the Routes tab.

XLSX export picked up a Fleet Mix block in the Network overview sheet — PVR-weighted propulsion split for the currently visible routes, matching what's on screen.

On the route card: a direction toggle in single-route mode flips outbound ⇄ inbound in place while the map re-renders in the new direction's colour. KPIs switched from Length to Stops, reading the pre-baked per-route count from route_stops.json.

v2.4 Apr 23, 2026
Bus-stop filter and a stops-first refresh

New in the filters panel: a Bus stop search, right below Propulsion. Pick a stop and the visible route set narrows to only the routes serving it — composed with every other filter (operator, route type, frequency, deck, propulsion) as AND. So "Go-Ahead electric routes passing through Camden Town" is one click away.

Under the hood, stops are now pre-baked weekly instead of fetched live from TfL. A new pipeline step produces data/stops.json (every London bus stop, with a reverse stop → routes index) and data/route_stops.json (per-route ordered stop lists with the "Towards …" label). Opening a route is instant, and the app behaves the same whether TfL is reachable or not.

Also: CI Actions bumped to v5 (Node 24) to clear the Node 20 deprecation warning, and the refresh pipeline grew from 8 to 9 steps to fit the new stops fetch.

v2.2 Apr 17, 2026
Accurate data: TfL-first pipeline with scrape fallbacks

Every route's operator, garage, vehicle type, PVR, deck, propulsion and destinations are now verified against the TfL Unified API, correcting routes that had been mis-assigned (e.g. route 1 had been tagged as a tram). The pipeline puts TfL first and uses londonbusroutes.net and bustimes.org only as fallbacks — TfL values are never overwritten by scraped ones.

New under the hood: numeric frequencies per peak band, a garages geodataset (postcodes.io geocoded), all London bus stops, and named bus / coach stations.

v2.0 Apr 15, 2026
Operator garages on the map and multi-sheet export

Operator garages on the map. Every London bus garage is now plotted at its real address, colour-coded by operator's brand livery. Hover for address, operator, and how many routes run from there.

Filters split by concern. Route type / frequency / deck / propulsion / operator drive the route lines. A separate Garage operator filter drives the garage markers independently — so one operator's route network can be overlaid on a competitor's garage footprint.

Smarter data pipeline. Night-route data (N128, N29, N9, …) that was previously blank now inherits from the daytime sibling when routes.htm confirms a 24-hour service. Unknown-operator routes dropped from 71 to 37 in a single refresh.

Export now produces a single .xlsx workbook with three sheets — Routes, Garages, and a Network overview — each honouring the currently applied filters.
